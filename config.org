* How this file works

This is a literate configuration: the guide to the system and its
implementation live in one document. =init.el= is only a bootstrap —
it calls =org-babel-load-file= on this file, which extracts every
=emacs-lisp= block (in order) into =config.el= and loads it.
Re-tangling happens only when =config.org= is newer than =config.el=,
so startup stays fast after the first run.

To change the configuration: edit the blocks here, restart Emacs (or
=M-x org-babel-load-file= on this file). Never edit =config.el= — it
is generated. In a team, this file lives in the shared repository: the
conventions that must be universal (keyword set, forced-reason states,
archive location) are distributed as code, not as a wiki page.

One standing rule from the leader-key mechanism: never bind anything
under =C-c= in this file — always bind under =M-m= directly.

* The One Round System

** The problem it solves

Plain-text project management conflates two kinds of information.
Records of moments — a daily note, a protocol, minutes — should never
be edited after the fact. Current truth — the state of a task, the
revision of a parameter, whether a project is alive — must evolve.
When both live in the same files, every change either corrupts a
record or goes unrecorded; reasons evaporate; deleted dead ends take
their lessons with them. The system below resolves this with one
mechanism that works identically for tasks, tickets, projects,
decisions, parameters, documents, and processes.

** The model — three shapes, one primitive

Every piece of information has one of three shapes, and the shape —
never the content type — determines its handling:

- A *record of a moment* (daily note, Laborprotokoll, minutes): true
  forever as written, never edited after its day, references
  everything else only by link.
- An *entity* (task, project, decision, parameter, document, process):
  has current truth that changes, plus a history of changes — with
  reasons — that must stay attached. Entities are born, change state,
  and die (finished, cancelled, or superseded), at which point they
  leave the workspace but not the record.
- An *understanding* (a concept note): your current best explanation,
  rewritten freely whenever you understand better; git keeps its
  history as a courtesy.

Everything rests on one primitive: *a timestamped, author-stamped,
reasoned log entry, attached to the thing that changed, written at the
moment of change.* Org provides it twice — automatically on state
changes (with reasons /forced/ where they matter), and manually via
=M-m M-z= for content edits. Delegation, supersession, document
revisions, and archiving are this primitive at different granularities.

** Folders are organizational defaults, not behavior

Every folder under =org/= is just a folder: it lives in the agenda, and
capture asks you which one to file into. The folders below exist only as
handy default targets and as archive homes — they no longer encode
behavior (agenda membership, mutability). A =#+filetags= entry marks the
/kind/ of content (=:labor:=, =:meeting:=, =:concept:=, …); one record
type or note type is one new template with its own filetag, never a new
folder.

| Folder     | Default use                                                                      |
|------------+----------------------------------------------------------------------------------|
| =inbox.org=  | unsorted front door (just one file you can pick during capture)                   |
| =dailies/=   | journal, one file per day; optional; never edited after its day                    |
| =records/=   | typed records (=:labor:=, =:meeting:=, …); immutable; link projects by ID          |
| =projects/=  | one file per project (the default for the =p= capture)                            |
| =areas/=     | machines, process domains, teams                                                  |
| =notes/=     | =:concept:= Zettel and =:collection:= items (movies, places — with TODO lifecycles) |
| =*/archive/= | every folder gets one; archive/ folders are the only ones NOT in the agenda       |

The agenda is =directory-files-recursively= over =org/= minus any
=archive/= folder, rebuilt before each =M-m a=. Add a folder and it shows
up; archive one (a subtree via =M-m x a=, or a whole file via =git mv=
into an =archive/=) and it drops out.

When a project changes something that belongs to an area (a machine
parameter, a standard process), the change happens *at the area* —
where current truth lives — with a log note linking the project that
caused it. Links are the joins; nothing is duplicated.

** What dailies are for (and when to skip them)

The daily note is the journal: the moment-record for everything that
has no other home. Because task creation and every state change are
already logged mechanically (=CREATED= stamps, =:LOGBOOK:= entries),
dailies carry *zero bookkeeping duty* — never write "opened task X"
into one. What remains for them is narrative: observations too small
for a Laborprotokoll, context ("Lieferant angerufen, Müller meldet
sich Montag"), half-thoughts, the informal counterpart to the formal
records. Their use case is answering, years later, "what was going on
around day X — what did I know, what was I trying?".

If you don't journal, skip dailies entirely: nothing else in the
system depends on them. They exist as capacity (=M-m n j=), not as
obligation.

** The five rituals

Everything you will ever do in this system is one of five moves:

1. *Capture* — =M-m c= (=t= todo, =n= note, =d= decision, =m= meeting,
   =l= Laborprotokoll, =p= project). Every template prompts for the
   folder (and file) under =org/= to file into; the typed name becomes
   the =#+title=. ID and =CREATED= stamp are automatic — every captured
   file gets a file-level ID, every captured/todo heading gets a heading
   ID, so every captured thing is a linkable roam node from birth. You
   never create entities by typing raw headings mid-thought — and that is
   what keeps moment-records safe: they only ever contain /links/ to
   entities, never the entities themselves.
2. *State change* — =M-m M-t=. Where a reason matters (WAIT, DLGT,
   CANCELLED, ONHOLD, SUPERSEDED, DROPPED), org refuses to proceed
   without one and stamps time + author into the =:LOGBOOK:=.
   Delegation is =M-m x g=: OWNER is set, the forced note is where you
   write the context the colleague needs.
3. *Change note* — =M-m M-z=. The manual twin of the state change: any
   meaningful content edit to an entity — a parameter value, a BOM
   row, a protocol step — gets a note saying WHAT changed and WHY.
   For document revisions, bump a =:REV:= property in the same breath
   and write the note in change-notice style: "Rev C: Pos 7 NBR →
   PEEK — NBR-Dichtungen versagen über 140 °C." The note is the
   searchable semantic layer; =git log -p= on the file is the
   full-text diff layer. Between the two, even a 400-row BOM has
   complete browsable history with zero snapshots in the workspace.
4. *Supersede* — =M-m x s=. When something is /replaced/ rather than
   revised or finished: decisions overruled, approaches abandoned,
   processes redesigned. Old and new are linked both ways by ID, the
   reason is forced, and the old node keeps its full content as a
   readable record until archived. =C-u M-m x s= starts the successor
   as a full copy — for document-style rework.
5. *Archive* — =M-m x a= for subtrees (→ =archive/= next to the file,
   stamped with time, origin, final state), =git mv= for whole project
   files. "Terminal" and "archivable" are separate judgements: a
   SUPERSEDED process revision stays in its area file while anyone
   might still operate under it.

The routing between 2, 4 and in-place revision comes down to two
distinctions. /Finished vs. replaced/: DONE and CANCELLED end a thing;
SUPERSEDED says "a successor exists" and points to it. /Revision vs.
supersession/: the same artifact evolving (BOM Rev B → Rev C) is an
in-place edit plus a change note, with git holding the old full text;
a different artifact taking its place (approach B replaces approach A)
is a supersession, where the old node remains readable forever.

#+begin_src org
,* SUPERSEDED Dichtungswerkstoff: NBR
:PROPERTIES:
:ID:            abc…
:SUPERSEDED_BY: [[id:xyz…]]
:END:

,* ACTIVE Dichtungswerkstoff: PEEK
:PROPERTIES:
:ID:         xyz…
:SUPERSEDES: [[id:abc…]]
:END:
#+end_src

** Dependencies and Gantt charts (org-edna)

Tasks gate each other through two properties, and every captured task
already carries the ID this needs (=M-x org-id-copy= on a heading puts
its ID on the clipboard):

- =:BLOCKER: ids(<id>)= — this task cannot be set DONE while the
  referenced task is still open. For simple chains, relative forms
  save the ID juggling: =:BLOCKER: previous-sibling= makes tasks
  depend on the one above them in the file.
- =:TRIGGER: ids(<id>) todo!(NEXT)= — when /this/ task is closed, the
  referenced task is automatically set to NEXT, so the next step
  surfaces in the agenda by itself. =next-sibling= works here too.

A typical project chain:

#+begin_src org
,* Aufgaben
,** TODO Prüfstand aufbauen
SCHEDULED: <2026-06-15 Mo> DEADLINE: <2026-06-19 Fr>
:PROPERTIES:
:ID:      aaa-1111-…
:TRIGGER: next-sibling todo!(NEXT)
:END:
,** TODO Messreihe fahren
:PROPERTIES:
:BLOCKER: ids("aaa-1111-…")
:EFFORT:  3d
:END:
#+end_src

The Gantt chart falls straight out of this: =M-m p g= in a project
file builds a Mermaid gantt from every task. =SCHEDULED= is the start,
=DEADLINE= the end (otherwise =:EFFORT:= like "3d" sets the duration,
default one day); =BLOCKER ids(…)= entries become =after=-dependencies,
so a task without its own SCHEDULED starts when its blockers end.
DONE/CANCELLED render greyed out, NEXT renders as active, priority
=[#A]= as critical, and top-level headings become sections. The export
writes =<project>.mmd= next to the org file and, when mermaid-cli is
installed (=npm i -g @mermaid-js/mermaid-cli=), renders and opens
=<project>.svg=.

** Time tracking — where the clock lives

Clock on the entity, never in the daily. =M-m i= on a task clocks in —
the clock lines land in /that task's/ =:LOGBOOK:=, exactly where the
rest of its history lives — =M-m o= clocks out, =M-m j= jumps to the
running clock, and =C-u M-m i= picks from recently clocked tasks, so
switching work is two keys from anywhere. The running clock shows in
the modeline and survives restarts.

Clocking into dailies would be an anti-pattern: it attributes time to
the /date/ instead of the /work/, and "wieviel Zeit hat Projekt X
gefressen" becomes unanswerable. The daily perspective on time is a
*view*, not a storage location: =M-m a d= shows today as a timeline of
all clocked intervals with a clock report on top — an automatically
written diary of the day. For per-project totals, drop a clocktable
into the project file and update it with =M-m M-c=:

#+begin_src org
,#+BEGIN: clocktable :scope file :maxlevel 3 :block thisweek :link t
,#+END:
#+end_src

If anything, this /reduces/ what dailies are for: the timeline of the
day writes itself, so the daily note stays purely narrative — what you
observed and thought, not what you did when.

** Working in a team

The repository is the team: one git repo holds the whole tree,
everyone clones it, pulls before the daily review, pushes after. The
one-file-per-project rule is also the merge-conflict strategy — people
on different projects never touch the same file, and when two people
do, logbook collisions are append-append conflicts (keep both lines,
done).

Delegation is ownership plus a reasoned state change. The task stays
in the project file where it belongs — tasks live with their project,
not with their person — and the colleague finds it in their
=M-m a m= ("My work") view after the next pull. Each person sets
=my/org-user= to match their git identity.

Colleagues without Emacs are degraded gracefully, not excluded: the
tree is plain text, any editor opens it, any project-wide search
replaces ripgrep, and the two human rules — never edit moment-records,
write a dated reason line when you change an entity — are followable
in Notepad. The tooling automates the conventions; it doesn't define
them.

Two conventions no tool can enforce, agree on them explicitly: reasons
are written *for the stranger* (the colleague three years from now who
finds the node via search — "Lieferantenproblem" helps nobody; "Müller
GmbH hat die 12mm-NBR-Variante eingestellt, keine Zweitquelle unter
5k MOQ" is gold), and the daily inbox review actually happens — an
inbox that silts up becomes a second workspace, and the system dies
of it.

** When to do what

| Situation                                        | Do this                                                                      |
|--------------------------------------------------+------------------------------------------------------------------------------|
| Anything incoming — idea, request, action needed | =M-m c t=; pick the folder/file; shape it at the daily review                  |
| What happened today, context, half-thoughts      | =M-m n j= daily note; link entities; never edit after today                  |
| Experiment                                       | =M-m c l= → typed record in =records/=, project linked by ID                 |
| Meeting                                          | =M-m c m=; decisions made there get =M-m c d= and are linked from the minutes |
| New task                                         | capture, then refile (=M-m M-w= — picks any node via roam) into the project  |
| Hand a task to a colleague                       | =M-m x g= — OWNER set, context note forced; appears in their My-work view    |
| Task no longer relevant                          | =M-m M-t c= — reason forced; =M-m x a= to archive                            |
| Machine parameter changes                        | edit the value in the area file; =M-m M-z=: alt → neu, warum, link the cause |
| Design decision made                             | =M-m c d= → refile to the project's Decisions                                |
| Decision overruled / approach replaced           | =M-m x s= on the old node; reason forced; write the successor                |
| Spec/BOM/protocol revision (same artifact)       | edit in place, bump =:REV:=, =M-m M-z= in change-notice style; =git log -p= has the diff |
| Document replaced by a different approach        | =C-u M-m x s= (successor = full copy), rework it                             |
| Project finished or cancelled                    | mark the heading (reason if cancelled); =git mv= to =projects/archive/=      |
| Learned something worth keeping                  | =M-m n c=, template =c= → concept note in =notes/=; rewrite freely forever   |
| Movie / place / item to track                    | =M-m n c=, template =k= → collection item in =notes/=                        |
| "Didn't we already try this once?"               | =M-m f g= from =~/org= — archives included; read the logbook you find        |

** The loop

Daily, ten minutes: pull; empty the inbox (refile, delegate, or
cancel-with-reason); optionally write the daily note; push. Weekly,
twenty minutes: sweep WAIT and DLGT (anything older than expected gets
a nudge or a state change with reason); archive everything terminal
whose dependencies are gone; skim ACTIVE decisions for any that
reality has silently overruled — supersede those /now/, while you
still remember why.

** Why this stays discoverable

The cancellation note you were forced to write contains the problem's
own vocabulary, so future searches for the problem find the abandoned
attempt. Supersession links form chains walkable in both directions
from any point. The archives are ordinary org files under the same
root, so ripgrep and the org-roam graph reach them without knowing
they are archives — and because every record links its project by ID,
a project's backlink buffer lists every protocol and meeting ever
taken for it, even after the project itself was archived. Git
underneath remains what it is good at: full-text diffs of large
documents, disaster recovery, proof. The semantic history — what
changed, when, by whom, and why — lives in the org layer, where search
can reach it.

** Leader keys — M-m ≙ C-c, M-o ≙ C-x
M-m is bound to the *same keymap object* that C-c points to
(`mode-specific-map') and M-o to `ctl-x-map'. That way C-c/C-x can be
repurposed as copy/cut further down without losing the prefix maps, and
everything any package ever binds under C-c <letter> automatically shows
up under M-m. Mode-local C-c/C-x prefixes are folded in per buffer (see
the CUA section). MUST run before any `:bind ("M-m ...")' below.

#+begin_src emacs-lisp
(define-key global-map (kbd "M-m") mode-specific-map)
(define-key global-map (kbd "M-o") ctl-x-map)

;; RULE: never bind mode-local keys under "C-c ..." anywhere in this config
;; after this point — always bind them under "M-m ..." directly. A late
;; (define-key some-mode-map (kbd "C-c x") …) would recreate a local C-c
;; prefix and confuse the remapping in the CUA section.

;; The two mechanism helpers live up here ON PURPOSE: packages that load
;; during init (dirvish pulls in dired, which pulls in wdired) run
;; `with-eval-after-load' blocks that call them immediately.

(defun my/mirror-c-to-m (keymap)
  "Add M-<key> bindings for every C-<key> binding in KEYMAP."
  (map-keymap
   (lambda (key def)
     (when (and (integerp key) (>= key ?\C-a) (<= key ?\C-z)
                (not (memq key '(?\C-g ?\C-h ?\C-i ?\C-j ?\C-m))))
       (let ((m-key (+ (- key ?\C-a) ?\M-a)))
         (unless (lookup-key keymap (vector m-key))
           ;; Skip keys that can't be mirrored: M-<k> is stored as ESC <k>,
           ;; which clashes when the map binds ESC itself to a command
           ;; (wdired does, for example).
           (ignore-errors
             (define-key keymap (vector m-key) def))))))
   keymap))

(defun my/minor-cua-fix (map)
  "Move MAP's C-c prefix bindings under M-m and free C-c.
MAP is a minor-mode keymap. Its C-c submap is mirrored (C-<k> -> M-<k>)
and exposed at M-m; sequences not in the submap fall through to the
major-mode and global leader maps."
  (let ((cc (lookup-key map (kbd "C-c"))))
    (when (keymapp cc)
      ;; If the submap binds ESC directly to a command (wdired: C-c ESC =
      ;; abort), every M-<k> mirror would clash with it, because Emacs
      ;; stores M-<k> as ESC <k>. The binding always duplicates C-c C-k,
      ;; so drop it before mirroring:
      (let ((esc (lookup-key cc (kbd "ESC"))))
        (when (and esc (not (keymapp esc)))
          (define-key cc (kbd "ESC") nil)))
      (my/mirror-c-to-m cc)
      (define-key map (kbd "M-m") cc)
      (define-key map (kbd "C-c") nil))))
#+end_src

*** Themes: professional dark/light pair with a switcher
#+begin_src emacs-lisp
;; Dark:  doom-dark+      — port of VS Code's Dark+ (somber, desaturated)
;; Light: doom-one-light  — clean, Office-like
;; M-m T toggles between them; the choice persists across restarts.
;; (doom-gruvbox stays installed — add it to `my/themes' to cycle it too.
;;  Built-in `modus-operandi' is another very professional light option.)
(defvar my/themes '(doom-dark+ doom-one-light)
  "Themes `my/theme-toggle' cycles through. First entry is the default.")
(defvar my/theme-file (expand-file-name "current-theme" user-emacs-directory)
  "File remembering the last active theme between sessions.")

(defun my/load-theme (theme)
  "Load THEME exclusively and remember it for the next session."
  (mapc #'disable-theme custom-enabled-themes)
  (load-theme theme t)
  (with-temp-file my/theme-file (insert (symbol-name theme))))

(defun my/theme-toggle ()
  "Cycle through `my/themes' (dark <-> light)."
  (interactive)
  (let* ((cur (car custom-enabled-themes))
         (next (or (cadr (memq cur my/themes)) (car my/themes))))
    (my/load-theme next)
    (message "Theme: %s" next)))

(use-package doom-themes
  :custom
  (doom-themes-enable-bold t)
  (doom-themes-enable-italic t)
  :bind (("M-m T" . my/theme-toggle))
  :config
  (condition-case nil
      (my/load-theme
       (or (and (file-exists-p my/theme-file)
                (intern (string-trim
                         (with-temp-buffer
                           (insert-file-contents my/theme-file)
                           (buffer-string)))))
           (car my/themes)))
    (error (my/load-theme (car my/themes))))
  (doom-themes-org-config))

(defun my/dired-create (name)
  "Create NAME in the current dired directory.
A trailing / creates a directory, anything else an empty file.

  (interactive "sNew (end with / for a directory): ")
  (let ((path (expand-file-name name (dired-current-directory))))

      (make-directory (file-name-directory path) t)

return an id link to it, or \"\" for none. Ensures the chosen file has a
file-level ID (creating one if needed), so org-roam backlinks list every
record taken for the project."
  (let* ((files (my/org-agenda-files))
         (rel   (mapcar (lambda (f) (file-relative-name f (expand-file-name org-directory))) files))
         (choice (completing-read "Projekt (leer = keins): " rel nil nil)))
    (if (or (null choice) (string-empty-p choice))
        ""
      (let ((abs (expand-file-name choice org-directory)))
        (with-current-buffer (find-file-noselect abs)
          (save-excursion
            (goto-char (point-min))
            (let ((id (org-id-get-create)))
              (save-buffer)
              (format "[[id:%s][%s]]" id (file-name-base abs)))))))))

** One Round System — lifecycle commands
The five rituals and where they live:
  1. Capture       M-m c     every template asks which folder/file under org/
  2. State change  M-m M-t   reasons FORCED on WAIT/DLGT/CANCELLED/
                             ONHOLD/SUPERSEDED/DROPPED
  3. Change note   M-m M-z   org-add-note — the manual "commit message"
                             for content edits (parameter changed, BOM
                             row swapped, protocol step adjusted): write
                             WHAT and WHY; time + author are automatic.
                             Full-text diffs of big documents stay git's
                             job (M-o g, git log -p) — the note is the
                             searchable semantic layer on top.
  4. Supersede     M-m x s   replaced-by-something-else (decisions,
                             approaches, processes); C-u M-m x s makes
                             the successor start as a full COPY (for
                             document-style rework)
  5. Archive       M-m x a   subtree -> archive/ next to its file;
                             whole project files: git mv to
                             projects/archive/ instead
Delegation = ownership + reasoned state change:   M-m x g

#+begin_src emacs-lisp
;; Every capture becomes a node. EVERY captured file gets a file-level ID
;; (idempotent: returns the existing one if present) — the :PROPERTIES:
;; drawer at the top makes the file a linkable roam node and lets the
;; links it contains (e.g. the record's "Projekt ::" line) register as
;; backlinks on their targets. ENTRY captures (todo, decision, meeting)
;; additionally get ID + CREATED on the heading; PLAIN file captures
;; (Projekt, Laborprotokoll) also ID every TODO heading that came in via
;; the template. org-roam's own captures are skipped: roam manages those
;; IDs itself, and dailies would drown in property drawers.
(defun my/org-capture-stamp ()
  "Make the just-captured item a node (One Round System)."
  (unless (bound-and-true-p org-roam-capture--node)
    ;; Every file gets a file-level ID (the buffer is widened before this
    ;; hook runs, so point-min is the real file start):
    (save-excursion
      (goto-char (point-min))
      (org-id-get-create))
    (pcase (org-capture-get :type)
      ('entry
       (save-excursion
         (org-back-to-heading t)
         (org-id-get-create)
         (org-set-property "CREATED"
                           (format-time-string "[%Y-%m-%d %a %H:%M]"))))
      ('plain
       ;; TODO headings embedded in the template (e.g. a project's Aufgaben)
       ;; get IDs too, so every todo is linkable from birth:
       (org-map-entries
        (lambda ()
          (when (and (org-get-todo-state) (not (org-id-get)))
            (org-id-get-create)))
        nil 'file)))))
(add-hook 'org-capture-before-finalize-hook #'my/org-capture-stamp)

;; Manually typed TODO headings (added via M-RET / org-insert-todo-heading

;; covered — use a heading command, or M-x org-id-get-create on it.
(defun my/org-id-new-todo-heading ()

  (ignore-errors

      (org-id-get-create))))

(defun my/org-supersede (&optional copy)
  "Supersede the entry at point with a successor sibling.
With prefix argument COPY (\\[universal-argument]), the successor starts
as a full copy of the old subtree — for reworking document-like entities.
Without it, the successor starts empty with the same title — for
decisions and approaches, where the new content is genuinely new.
Links predecessor and successor in both directions via ID, then marks
the old entry SUPERSEDED; org asks for the reason at the end (forced)."
  (interactive "P")
  (org-back-to-heading t)
  (let ((lvl    (org-outline-level))
        (old-id (org-id-get-create))
        (title  (org-get-heading t t t t))
        new-id)
    (if copy
        (progn
          (org-copy-subtree)
          (org-end-of-subtree t t)
          (let ((pos (point)))
            (org-paste-subtree lvl)
            (goto-char pos))
          (org-delete-property "ID"))
      (org-insert-heading-after-current)
      (insert title))
    (setq new-id (org-id-get-create))
    (org-set-property "SUPERSEDES" (format "[[id:%s]]" old-id))
    (org-entry-put nil "TODO" "ACTIVE")
    (save-excursion
      (org-id-goto old-id)
      (org-set-property "SUPERSEDED_BY" (format "[[id:%s]]" new-id))
      (org-todo "SUPERSEDED"))))

(defun my/org-delegate (person)
  "Hand the entry at point to PERSON: set OWNER, then enter DLGT.
The forced note is where the context goes that PERSON will need —
write it for them, not for yourself."
  (interactive "sDelegieren an: ")
  (org-set-property "OWNER" person)
  (org-todo "DLGT"))

*** Fix: seconds-to-string arity
#+begin_src emacs-lisp
;; Marginalia (and org-timeblock) call `seconds-to-string' with 3 args,
;; but the built-in subr only accepts 1. Override early so it accepts extras.
;; Redefine only when the current definition accepts FEWER than the 3 args
;; marginalia passes (newer Emacs versions already accept them natively).
;; time-date must be loaded first — on an autoload stub, `func-arity' errors:
(require 'time-date)
(let ((max-arity (cdr (func-arity (symbol-function 'seconds-to-string)))))
  (when (and (numberp max-arity) (< max-arity 3))
    (defun seconds-to-string (delay &rest _ignored)
    "Convert the time DELAY, in seconds, to a string."
    (cond ((< delay 0) (format "-%s" (seconds-to-string (- delay))))
          ((>= delay (* 1 60 60 24 365.25))
           (format "%.1fy" (/ delay (* 1 60 60 24 365.25))))
          ((>= delay (* 1 60 60 24 30.4375))
           (format "%.1fmo" (/ delay (* 1 60 60 24 30.4375))))
          ((>= delay (* 1 60 60 24))
           (format "%.1fd" (/ delay (* 1 60 60 24))))
          ((>= delay (* 1 60 60))
           (format "%.1fh" (/ delay (* 1 60 60))))
          ((>= delay (* 1 60))
           (format "%.1fm" (/ delay (* 1 60))))
          (t (format "%.1fs" delay))))))

** Gantt export: org + org-edna -> Mermaid
M-m p g in any org buffer. Builds a Mermaid gantt chart from all TODO
headlines: SCHEDULED = start, DEADLINE = end (else :EFFORT:, default 1d),
org-edna `:BLOCKER: ids(...)' entries become mermaid `after' dependencies,
DONE/NEXT/[#A] map to done/active/crit styling. Top-level headings become
sections. Renders to SVG if mmdc (mermaid-cli) is installed, otherwise
writes the .mmd source (also usable in a babel block / Marginalia app).

#+begin_src emacs-lisp
(defun my/org-gantt--date (ts)
  "Extract YYYY-MM-DD from an org timestamp string TS, or nil."
  (when (and ts (string-match "[0-9]\\{4\\}-[0-9]\\{2\\}-[0-9]\\{2\\}" ts))
    (match-string 0 ts)))

(defun my/org-gantt--days (effort)
  "EFFORT string (\"3d\", \"2:00\") -> whole days, minimum 1."
  (or (when effort
        (ignore-errors
          (max 1 (round (/ (org-duration-to-minutes effort) 1440.0)))))
      1))

(defun my/org-gantt--blockers (blocker)
  "Extract the list of ids from an org-edna BLOCKER property string."
  (when (and blocker (string-match "ids(\\([^)]*\\))" blocker))
    (split-string (match-string 1 blocker) "[ \t,\"]+" t)))

(defun my/org-gantt--clean (name)
  "Strip characters mermaid chokes on in task NAMEs."
  (replace-regexp-in-string "[:,#]" "-" name))

(defun my/org-mermaid-gantt ()
  "Return Mermaid gantt source for all TODO headlines in the current buffer."
  (require 'org-duration)
  (let ((ids (make-hash-table :test #'equal)) ; org ID/CUSTOM_ID -> mermaid id
        (n 0) entries)
    ;; Pass 1: collect tasks, assign short mermaid ids.
    (org-map-entries
     (lambda ()
       ;; Tasks only: the artifact states (ACTIVE/ONHOLD/FINISHED/
       ;; SUPERSEDED/DROPPED — decisions, documents, processes) carry no
       ;; schedule semantics and stay out of the gantt.
       (when (member (org-get-todo-state)
                     '("TODO" "NEXT" "WAIT" "DLGT" "DONE" "CANCELLED"))
         (let* ((oid (or (org-entry-get nil "ID")
                         (org-entry-get nil "CUSTOM_ID")))
                (mid (format "t%d" (setq n (1+ n)))))
           (when oid (puthash oid mid ids))
           (push (list :mid mid
                       :name (my/org-gantt--clean (org-get-heading t t t t))
                       :todo (org-get-todo-state)
                       :prio (org-entry-get nil "PRIORITY")
                       :sched (my/org-gantt--date (org-entry-get nil "SCHEDULED"))
                       :dead  (my/org-gantt--date (org-entry-get nil "DEADLINE"))
                       :days  (my/org-gantt--days (org-entry-get nil "EFFORT"))
                       :deps  (my/org-gantt--blockers (org-entry-get nil "BLOCKER"))
                       :section (or (car (org-get-outline-path)) "Tasks"))
                 entries))))
     t nil)
    (setq entries (nreverse entries))
    ;; Pass 2: emit.
    (let ((title (my/org-gantt--clean
                  (or (cadr (assoc "TITLE" (org-collect-keywords '("TITLE"))))
                      (and buffer-file-name (file-name-base buffer-file-name))
                      "Project"))))
      (with-temp-buffer
        (insert "gantt\n"
                "  dateFormat YYYY-MM-DD\n"
                (format "  title %s\n" title))
        (let ((cur-section nil))
          (dolist (e entries)
            (unless (equal cur-section (plist-get e :section))
              (setq cur-section (plist-get e :section))
              (insert (format "  section %s\n" (my/org-gantt--clean cur-section))))
            (let* ((todo (plist-get e :todo))
                   (flags (concat
                           (cond ((member todo '("DONE" "CANCELLED")) "done, ")
                                 ((equal todo "NEXT") "active, ")
                                 (t ""))
                           (when (equal (plist-get e :prio) "A") "crit, ")))
                   (deps (delq nil (mapcar (lambda (d) (gethash d ids))
                                           (plist-get e :deps))))
                   (start (cond ((plist-get e :sched) (plist-get e :sched))
                                (deps (concat "after " (mapconcat #'identity deps " ")))
                                (t (format-time-string "%Y-%m-%d"))))
                   (end (or (plist-get e :dead)
                            (format "%dd" (plist-get e :days)))))
              (insert (format "  %s :%s%s, %s, %s\n"
                              (plist-get e :name) (or flags "")
                              (plist-get e :mid) start end)))))
        (buffer-string)))))

(defun my/org-gantt-export ()

*** Browser-style buffer cycling (C-Tab / C-S-Tab)
#+begin_src emacs-lisp
;; Ctrl+Tab / Ctrl+Shift+Tab flip through open buffers like browser tabs.
;; Hold Ctrl and tap Tab to step forward, add Shift to step back. Each
;; tap redraws a one-line "tab strip" in the echo area: the current
;; buffer is highlighted and first (the rest follow in most-recent-first
;; order), and the strip auto-clears ~1.5 s after you stop. Every open
;; buffer is included.
;;
;; Why no transient keymap: every cycling key is a real C-Tab/C-S-Tab, so
;; it hits the global binding directly and plain Tab is never hijacked.
;; `next-buffer'/`previous-buffer' use Emacs's per-window cycle list, so
;; taps traverse A->B->C->A without bouncing back and forth.
(defvar my/cycle--timer nil)

(defun my/cycle--strip ()
  "MRU buffer names as a tab strip; current buffer highlighted and first."
  (let ((cur (current-buffer)))
    (mapconcat
     (lambda (b)
       (let ((n (buffer-name b)))
         (if (eq b cur)
             (propertize (concat "[ " n " ]") 'face 'success)
           (propertize n 'face 'shadow))))
     (buffer-list) "  ")))

(defun my/cycle--echo ()
  "Show the buffer-cycle tab strip in the echo area, auto-hide after 1.5 s."
  (message "%s" (my/cycle--strip))
  (when my/cycle--timer (cancel-timer my/cycle--timer))
  (setq my/cycle--timer (run-with-timer 1.5 nil (lambda () (message nil)))))

(defun my/cycle-next-buffer (&optional n)
  "Cycle to the next buffer (browser Ctrl+Tab)."
  (interactive "p")
  (next-buffer (or n 1))
  (my/cycle--echo))
(defun my/cycle-prev-buffer (&optional n)
  "Cycle to the previous buffer (browser Ctrl+Shift+Tab)."
  (interactive "p")
  (previous-buffer (or n 1))
  (my/cycle--echo))

(global-set-key (kbd "C-<tab>")           #'my/cycle-next-buffer)

;; forms you might see in C-h k, then trim to the one that fires:
(global-set-key (kbd "C-S-<iso-lefttab>") #'my/cycle-prev-buffer)

*** Make Esc actually mean "cancel" like in other editors
#+begin_src emacs-lisp
;; By default Esc in Emacs is a Meta prefix. Most people coming from VS Code
;; expect Esc to abort selections / multi-cursor / minibuffer.
(defun my/keyboard-escape-quit ()
  "Cancel current action: minibuffer, region, multiple cursors, etc."
  (interactive)
  (cond
   ;; In the minibuffer? Close it.
   ((active-minibuffer-window) (abort-recursive-edit))
   ;; Multi-cursor mode active? Leave it and clear all leftover regions.
   ((bound-and-true-p multiple-cursors-mode)
    (multiple-cursors-mode 0)
    (deactivate-mark)
    ;; mc/keyboard-quit also clears the fake-cursor overlays cleanly:
    (when (fboundp 'mc/keyboard-quit) (mc/keyboard-quit)))
   ;; Just a region? Deselect it.
   ((use-region-p) (deactivate-mark))
   ;; Fallback: standard abort.
   (t (keyboard-quit))))
(global-set-key (kbd "<escape>") #'my/keyboard-escape-quit)
